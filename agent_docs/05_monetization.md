# 05. マネタイズ設計

## 訴求軸

**「決められない、からの卒業。」**

ユーザーが自分ごととして読める言葉で、Troisが解く課題を一言で表す。
「AIが店を探します」ではなく、感情・体験への訴求にする。
App Store サブタイトル・プロモーション文言の軸として使う。

---

## 基本方針

**フリーミアム × サブスク**モデルを採用する。

Anthropic Claude API は呼び出しごとに費用が発生する変動費モデルのため、
使用量に応じた収益が見込めるサブスクが最も相性がよい。
買い切りはヘビーユーザーほど赤字リスクが高く、広告モデルは「迷いをなくす」というUXと相反する。

課金実装は MVP 検証後に行うが、**公開時には課金実装済みの状態でリリースする**。
まず「3件に絞ることが本当に喜ばれるか」を検証することが最優先。

## プラン設計

| プラン | 価格 | 利用上限 |
|---|---|---|
| 無料 | 無料 | 累計30回まで |
| 月額サブスク | 500円 / 月 | 1日10回まで |
| 年額サブスク | 4,800円 / 年（2ヶ月分お得） | 1日10回まで |

### 無料枠を30回にした理由
- 習慣形成には最低1〜2週間の継続利用が必要
- 10回では早すぎて「ないと困る」体験に至る前に終わってしまう
- 昼だけ毎日使っても約1ヶ月使えるため、習慣化を十分に促せる
- 外国人観光客など短期利用者もこの枠内でカバーできる
- 短期プランは App Store の課金設計が複雑になるため設けない

### 1日上限を設ける理由
- 「使い放題」にすると API コストが青天井になるリスクがある
- 1日10回あれば実用上は十分で、APIコストも予測しやすくなる

### 価格の根拠
- 食べログプレミアム（約400円）・乗り換えアプリ系（300〜500円）が相場
- 「決断を買うアプリ」というポジションは価格耐性が高い
- まず500円で出し、コンバージョン率を見ながら調整する

## API コスト試算

**採用モデル: Claude Haiku 4.5**（Sonnet と品質比較検証後に確定）
- 推薦理由の生成は高度な推論不要のため Haiku で十分と判断
- Sonnet 比で約1/10のコストになる

**1回の呼び出しあたりのトークン数（目安）**

| 内容 | トークン数 |
|---|---|
| システムプロンプト | 約300〜500 |
| 候補店リスト（20軒分の構造化データ） | 約800〜1,500 |
| ユーザー条件 | 約100 |
| 出力（3軒 × 理由2文） | 約200〜400 |
| **合計** | **約1,500〜2,500** |

**1回あたりのコスト（Haiku 使用時）**
- 1回あたり約 **0.07〜0.15円**

**無料30回を使い切った場合の持ち出し（1ユーザーあたり）**
- 30回 × 0.15円 = 約 **4〜5円**
- 1,000人が無料枠を使い切っても約4,000〜5,000円 → 検証コストとして許容範囲

**コスト削減策**
- **プロンプトキャッシュ**（Anthropic の Prompt Caching）を使うと、システムプロンプト分の入力コストをさらに削減できる。実装時に必ず組み込む。

---

## 実装方針

- **StoreKit 2**（iOS標準）でサブスク課金を実装
- 利用回数カウントは **iCloud（NSUbiquitousKeyValueStore）** に保存し、端末をまたいで管理
  - iCloud が使えない場合は UserDefaults にフォールバック
- 年額プランを併設することで、キャッシュフローの安定化を図る
