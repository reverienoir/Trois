# 05. マネタイズ設計

## 基本方針

**フリーミアム × サブスク**モデルを採用する。

Anthropic Claude API は呼び出しごとに費用が発生する変動費モデルのため、
使用量に応じた収益が見込めるサブスクが最も相性がよい。
買い切りはヘビーユーザーほど赤字リスクが高く、広告モデルは「迷いをなくす」というUXと相反する。

---

## プラン設計

| プラン | 価格 | 利用上限 |
|---|---|---|
| 無料 | 無料 | 累計10回まで |
| 月額サブスク | 500円 / 月 | 1日10回まで |
| 年額サブスク | 4,800円 / 年（2ヶ月分お得） | 1日10回まで |

### 無料枠の考え方
- 10回という明確な数字で、ユーザーが価値を実感してから課金判断できる
- 「迷いを減らす」アプリの性質上、ハマった人ほど繰り返し使う → サブスクへの自然な誘導になる
- 外国人観光客など短期利用者は無料10回で十分カバーできる（数日の観光で毎食使っても10回以内）
  - 短期プランは App Store の課金設計が複雑になるため設けない

### 1日上限を設ける理由
- 「使い放題」にすると API コストが青天井になるリスクがある
- 1日10回あれば実用上は十分で、APIコストも予測しやすくなる

### 価格の根拠
- 食べログプレミアム（約400円）・乗り換えアプリ系（300〜500円）が相場
- 毎日ランチに使う場合、1回あたり約17円 → 時間節約に対して払う価値がある
- 「決断を買うアプリ」というポジションは価格耐性が高い
- まず500円で出し、コンバージョン率を見ながら調整する

## API コスト試算

モデル: Claude Sonnet 4.6

**1回の呼び出しあたりのトークン数（目安）**

| 内容 | トークン数 |
|---|---|
| システムプロンプト | 約300〜500 |
| 候補店リスト（20軒分の構造化データ） | 約800〜1,500 |
| ユーザー条件 | 約100 |
| 出力（3軒 × 理由2文） | 約200〜400 |
| **合計** | **約1,500〜2,500** |

**1回あたりのコスト**
- 入力: $3 / 100万トークン、出力: $15 / 100万トークン
- 1回あたり約 **$0.005〜$0.01**（0.7〜1.5円）

**月額500円ユーザーが1日10回使い続けた場合**
- 月300回 × 1.5円 = 約450円 → ギリギリ赤字になりうる
- → 1日上限の設定はコスト管理上も必須

**コスト削減策**
- **プロンプトキャッシュ**（Anthropic の Prompt Caching）を使うと、システムプロンプト分の入力コストを大幅削減できる。実装時に必ず組み込む。

## 実装方針

- **StoreKit 2**（iOS標準）でサブスク課金を実装
- 利用回数カウントは **iCloud（NSUbiquitousKeyValueStore）** に保存し、端末をまたいで管理
  - iCloud が使えない場合は UserDefaults にフォールバック
- 年額プランを併設することで、キャッシュフローの安定化を図る
