# 04. 技術アーキテクチャ / データフロー / API候補

## 全体データフロー

```
[1] 現在地取得            CoreLocation で緯度経度を取得
        |
[2] 条件入力              徒歩分数・予算・ジャンル（HomeView）
        |
[3] 候補店の取得          飲食店API（ホットペッパー等）に範囲・ジャンル・予算で問い合わせ
        |                 → 候補リスト（数件〜数十件）
        |
[4] AIで3軒に選定          候補リスト＋ユーザー条件を Claude API に渡し、
        |                 「3軒＋各推薦理由」を生成
        |
[5] 表示                  ResultView に3軒のカードを表示
```

ポイント: **「探す」のは API、「選んで理由をつける」のが AI**。
この役割分担がアプリの核（＝決断補助）を担う。

---

## 飲食店データソース候補

### 第一候補: ホットペッパーグルメ サーチAPI（リクルートWEBサービス）
- エンドポイント: `https://webservice.recruit.co.jp/hotpepper/gourmet/v1/`
- **無料**。リクルートWEBサービスに登録して **APIキー** を取得して利用。
- 緯度経度 + `range`（300m〜3000mの5段階）+ ジャンル + 予算で検索でき、
  本アプリの「徒歩◯分・ジャンル・予算」とそのまま対応する。
- レスポンス: JSON / XML 選択可。

**制約（必ず守る）:**
- **クレジット表記が必須**（利用するアプリにロゴ or テキストでクレジットを表示）。ResultView 等に入れる。
- 取得した **店舗情報そのものの第三者再販は禁止**。
- 予算は **ディナー予算** が基本（ランチ予算は持たない）。
- 旧API（`api.hotpepper.jp`）は提供終了済み。現行の `webservice.recruit.co.jp` を使う。
- ※ 最新の利用規約・項目は公式ドキュメントで都度確認すること。

### 代替候補
- **Google Places API**: 全世界対応・情報豊富だが従量課金。海外展開を見据えるなら。
- **ぐるなび API 等**: 提供状況・規約を要確認。

> MVP は日本国内・無料運用しやすさを優先し **ホットペッパー第一候補** で進めるのが現実的。

## AI 推薦の設計（Claude API）

- 入力: 候補店リスト（名前・ジャンル・予算・距離・評価などの構造化データ）＋ ユーザー条件。
- 出力: **ちょうど3軒** の選定と、各店の **推薦理由（1〜2文）**。
- 出力は構造化（JSON）で受けると扱いやすい。
  - 例: `[{ "id": ..., "reason": ..., "role": "本命/穴場/無難" }, ...]` を返すようプロンプトで指定し、パースする。
- プロンプト設計の指針:
  - 「3軒に必ず絞る」「各店になぜそれを選んだか理由を必ずつける」を明示。
  - ユーザーの気分・予算・徒歩条件を尊重し、外れた店は出さない。
  - 候補が3軒未満のときの振る舞い（あるだけ出す＋不足を伝える）も定義。

## 注意・検討事項（実装前に決めておく）

- **APIキーの管理**: ホットペッパー / Claude のキーをコードに直書きしない。
  個人開発でもクライアント直叩きはキー露出リスクがあるため、
  必要なら軽量バックエンド（Firebase Functions 等）を挟むか、配布形態に応じて方式を決める。
- **課金 / レート制限**: Claude API は呼び出しごとにコスト発生。
  1リクエストにつき候補をまとめて1回で選定させ、呼び出し回数を抑える設計に。
- **プライバシー**: 位置情報は利用目的を明示し、必要時のみ取得。Info.plist の用途文言を用意。
- **オフライン / 失敗時**: API・AIの失敗時のエラー表示とリトライ導線。
- **クレジット表記**: ホットペッパー利用時の表記を ResultView に組み込む（前述）。
